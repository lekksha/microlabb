# Отчёт по лабораторной работе
## Разработка и рефакторинг микросервисного приложения VegasShop

---

## 1. Описание системы

**VegasShop** — платформа интернет-магазина, реализованная в виде набора независимых микросервисов. Система обеспечивает регистрацию и аутентификацию пользователей, просмотр товаров по магазинам, оформление заказов, ведение истории покупок, периодическое пополнение склада и хранение файлов пользователей.

Технологический стек:
- **Язык и фреймворк:** C# / ASP.NET Core 3.1
- **База данных:** Microsoft SQL Server 2019 (отдельная БД на каждый сервис)
- **Брокер сообщений:** RabbitMQ 3.11
- **Сервисная шина:** MassTransit 7.1.6
- **API-шлюз:** Ocelot 17.0.0
- **Контейнеризация:** Docker / Docker Compose
- **Аутентификация:** JWT (JSON Web Token)
- **Фронтенд:** HTML5 + Vanilla JavaScript, обратный прокси — Nginx

## 2. Архитектурные компоненты

### 2.1 Балансировщик нагрузки (Nginx)

Два идентичных контейнера `nginx1` (порт 8081) и `nginx2` (порт 8082) обслуживают клиентские запросы и перенаправляют их к API-шлюзу. Наличие двух экземпляров обеспечивает горизонтальное масштабирование на уровне представления.

### 2.2 Фронтенд (Frontend)

Одностраничное приложение на HTML + JavaScript. Функциональность:
- Аутентификация и регистрация пользователей
- Просмотр магазинов и товаров с фильтром по категориям
- Корзина с накоплением позиций и оформлением заказа
- История покупок с возможностью редактирования типа оплаты
- Файловое хранилище: загрузка, просмотр и скачивание файлов

### 2.3 API-шлюз (VegasShop.ApiGateway, порт 5000)

Единая точка входа на базе **Ocelot**. Маршрутизирует все входящие HTTP-запросы к соответствующим микросервисам:

| Upstream-путь | Сервис | Порт |
|---|---|---|
| `/api/account/*` | Identity | 7001 |
| `/api/purchases/*` | Purchases | 7002 |
| `/api/shops/*` | Shops | 7003 |
| `/api/factories/*` | Factories | 7004 |
| `/api/files/*` | Files | 7005 |

### 2.4 Брокер сообщений (RabbitMQ)

Асинхронный обмен сообщениями между сервисами через очереди:
- `identityQueue` — аутентификация
- `purchasesQueue` — транзакции покупок
- `shopsQueue` — управление товарами

### 2.5 База данных (SQL Server 2019)

Каждый сервис имеет **собственную базу данных** (паттерн Database per Service):

| Сервис | БД |
|---|---|
| Identity | `identity` |
| Purchases | `purchases` |
| Shops | `shops` |
| Factories | `factories` |

### 2.6 Микросервисы

#### Identity (порт 7001)
Управление пользователями на базе ASP.NET Core Identity. Регистрация, вход, валидация JWT-токена. Использует MassTransit-consumers для внутренних запросов от других сервисов.

#### Purchases (порт 7002)
Хранение и управление историей покупок пользователей. Получает события создания транзакций от Shops-сервиса через RabbitMQ. Поддерживает типы оплаты: наличные (InCash) и карта (ByCard).

#### Shops (порт 7003)
Управление магазинами и товарами. Обрабатывает заказы (уменьшение остатков), создаёт транзакцию и отправляет уведомление в Purchases. Получает пополнение склада от Factories.

#### Factories (порт 7004)
Фоновый сервис. Каждые 30 секунд автоматически публикует в `shopsQueue` запрос на пополнение остатков товаров.

#### Files (порт 7005) — *новый сервис*
Файловое хранилище для бинарных данных пользователей. Сохраняет файлы в Docker-volume `files_data`. Endpoints: загрузка (`POST /api/files/upload`), получение (`GET /api/files/{fileId}`), список (`GET /api/files/`), удаление (`DELETE /api/files/{fileId}`).

## 3. Межсервисное взаимодействие

Используются два вида взаимодействия:

**Синхронное (HTTP через API-шлюз):** клиент → Nginx → API Gateway → микросервис.

**Асинхронное (RabbitMQ + MassTransit):**

```
Клиент
  └─► Shops.API ─[shopsQueue]─► Shops.Consumer(BuyProducts)
                                      │
                                      └─[purchasesQueue]─► Purchases.Consumer(AddTransaction)

Factories.API (фоновый)
  └─[shopsQueue]─► Shops.Consumer(AddProductsByFactory)
```

## 4. Соответствие нефункциональным требованиям

### 4.1 Непрерывная работа и устойчивость к сбоям
- Политика `restart: unless-stopped` для всех контейнеров
- Healthcheck-проверки: каждые 15 с, до 5 попыток
- Зависимости `depends_on: condition: service_healthy` — сервис стартует только после готовности зависимостей
- Два экземпляра фронтенда (`nginx1`, `nginx2`) обеспечивают отказоустойчивость на уровне представления

### 4.2 Обновление без остановки сервиса
- Контейнерная архитектура позволяет перезапускать каждый сервис независимо
- Остальные сервисы продолжают работать через брокер и шлюз

### 4.3 Масштабируемость
- Изолированные сети Docker (`frontend`, `gateway`, `backend`, `messaging`) исключают лишние зависимости
- Каждый микросервис масштабируется независимо (можно добавить реплики через `scale`)
- Отдельные БД исключают конкуренцию за ресурсы

### 4.4 Функциональная гетерогенность
| Функция | Сервис | Технология |
|---|---|---|
| Аутентификация | Identity | ASP.NET Identity + JWT |
| Транзакции | Purchases | EF Core + SQL Server |
| Магазин | Shops | EF Core + RabbitMQ |
| Склад | Factories | Hosted Service (фоновый) |
| Файлы | Files | Filesystem + Docker Volume |
| Маршрутизация | ApiGateway | Ocelot |

## 5. Соответствие признакам микросервисной архитектуры

| Признак | Реализация |
|---|---|
| Слабая связанность | Сервисы общаются через HTTP (Gateway) и сообщения (RabbitMQ), прямых вызовов нет |
| Изоляция технологий | Каждый сервис — независимый проект со своим стеком |
| Сетевое взаимодействие | REST API через Ocelot + MassTransit через RabbitMQ |
| Контейнерный дизайн | Каждый сервис — отдельный Docker-контейнер с Dockerfile |
| Географическое распределение | Независимые контейнеры готовы к развёртыванию на разных узлах |

## 8. Описание выполненных изменений

### 8.1 Переименование пространств имён
Все пространства имён и сборки переименованы с `RtuItLab.*` на `VegasShop.*`:
- `VegasShop.Infrastructure` — общая инфраструктурная библиотека
- `VegasShop.ApiGateway` — API-шлюз
- `src/VegasShop.sln` — файл решения

Затронуто: ~90 файлов (`.cs`, `.csproj`, `.sln`, `Dockerfile`, `.json`, `.md`, `.yml`).

### 8.2 Новый микросервис Files.API
Добавлен сервис файлового хранилища:
- `POST /api/files/upload` — загрузка файла, возвращает уникальный идентификатор
- `GET /api/files/{fileId}` — скачивание файла
- `GET /api/files/` — список загруженных файлов
- `DELETE /api/files/{fileId}` — удаление файла
- Файлы хранятся в Docker-volume `files_data` по пути `/app/uploads`
- Маршруты добавлены в `ocelot.json`; сервис зарегистрирован в `docker-compose.yml`

### 8.3 Улучшение фронтенда
- Переименован в **VegasShop**
- Добавлена **корзина**: кнопка «В корзину» у каждого товара, боковая панель с позициями, итоговой суммой и оформлением заказа
- Добавлен **фильтр по категории** товаров (вызывает `/api/shops/{id}/find_by_category`)
- Добавлена вкладка **«Файлы»**: загрузка через `<input type="file">`, список файлов с кнопкой скачивания и удаления
- Улучшено отображение покупок: тип оплаты (Наличные/Карта), итоговая сумма транзакции
- Добавлена защита от XSS (функция `escHtml`)
- Отображение низкого остатка товара красным цветом
